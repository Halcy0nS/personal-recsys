# 产品路线图

[English](product_roadmap.md)

## 目的

这份文档是面向长期产品演进的路线图，用来说明如何把当前仓库逐步做成一个**本地单用户、可交互、推荐质量达标**的产品。

它和其他文档的角色不同：

- `README_ZH.md` 说明当前代码今天已经实现了什么
- `docs/PROJECT_STATUS.md` 说明当前实现和研究进展
- `docs/project_routes_zh.md` 说明当前仓库实际上有哪些路线，以及哪些是主线/支撑线
- 本文档说明接下来按什么顺序建设产品，以及每个阶段的目标是什么

这份路线图建立在当前代码事实之上：

- 运行时推荐核心在 `src/`
- 对外主入口是 `src.engine.PersonalRecSys`
- rerank、稳定性分析和 simulator 评测能力已经存在，但目前仍然分散在运行时模块和评测模块中

## 产品目标

长期目标不是继续堆实验脚本，而是把仓库演进成一个具备以下特征的产品：

- 推荐质量可靠
- 结果可解释
- rerank 能力可复用
- 本地单用户交互流程完整
- 可以被重复使用，而不是一次性跑通

默认规划假设：

- 本地单用户优先
- 推荐质量优先
- 排序核心可信后再补交互层
- 不提前按多用户服务平台来设计

## 基本原则

1. 产品力先建立在推荐质量上。
2. 运行时能力应沉到 `src/`，不能长期停留在 `src/evaluation`。
3. 评测工具仍然重要，但它们服务于产品迭代，而不是直接等于产品本身。
4. 公开 API 优先通过“只做加法”的方式演进。
5. 文档必须明确区分“当前已实现”与“未来计划”。

## Phase 1：推荐核心质量底座

目标：把主推荐链路补成一个完整、稳定、可解释的推荐引擎。

重点工作：

- 统一配置优先级和真实生效路径
- 明确 `recommend()` 的正式运行边界
- 把 `min_score`、`popularity`、候选元数据等已有能力更完整地打通到主链
- 真正实现 `RECENCY`、`DIVERSITY`，而不是只停留在枚举里
- 增强结果解释能力，让每个排序结果都能稳定暴露分数组成

验收标准：

- 不依赖 `src/evaluation` 也能独立跑通完整推荐流程
- 推荐结果可解释，配置变化带来的行为变化是可预期的
- 脚本式测试能覆盖主链最主要的运行行为

## Phase 2：通用 Rerank 插槽产品化

目标：把 rerank 从“点式实验能力”升级为主链的一等扩展点。

重点工作：

- 在 `src/contracts` 中定义通用 reranker 抽象
- 标准化 rerank 输出结构，让 pointwise 和后续 RAG rerank 共用运行时契约
- 保持 `PipelineOrchestrator` 作为主编排中心
- 让 `PersonalRecSys` 支持以加法方式注入、切换和查看 reranker
- 把已经验证过的 rerank 实现从 evaluation 语义里剥离出来，迁移到运行时更合适的位置

验收标准：

- 默认 pointwise rerank 能通过主引擎正式启用
- candidate-aware RAG rerank 能通过同一个运行时插槽接入
- 不再需要依赖实验脚本才能把 rerank 接到主推荐流程里

## Phase 3：把评测体系沉淀为产品研发基础设施

目标：保留研究能力，但让它们真正服务产品迭代。

重点工作：

- 持续保留并完善：
  - reranker stability
  - order instability attribution
  - simulator Glicko2 / fixed-pair 校准
- 让主链输出与评测输入的数据结构更容易互通
- 用评测来回答产品研发问题，例如：
  - 这次排序改动是真的更好，还是随机波动？
  - 新 reranker 是否不稳定？
  - 观察到的提升是不是 judge 偏差？

验收标准：

- 主要排序改动都能通过统一评测脚本做对比
- 研发过程能明确区分“质量提升”“随机波动”“judge 偏差”
- simulator 结果始终以内部合成评测产物来表述，不冒充真人真值

## Phase 4：补齐单用户交互闭环

目标：把推荐引擎真正暴露成一个可用的本地工作流。

重点工作：

- 支持最核心的用户闭环：
  - 导入收藏
  - 构建并查看画像
  - 导入候选内容
  - 运行推荐
  - 查看推荐解释
  - 可选比较 baseline 与 rerank 的差异
- 交互层第一版可以很轻，但必须围绕真实用户任务设计，而不是围绕实验脚本设计

验收标准：

- 非开发者在不改源码的前提下，也能完成一轮完整推荐流程
- 推荐结果、画像摘要和解释信息能被直接阅读和比较
- 交互层反映的是产品使用路径，而不是内部实验流程

## Phase 5：产品级工程打磨

目标：让本地产品稳定、可理解、可重复使用。

重点工作：

- 配置管理
- 缓存一致性
- 长任务进度可观测性
- 日志与失败恢复
- 更清晰的数据与产物目录规范
- 环境检查、示例和文档维护

验收标准：

- 冷启动、重复运行、重启恢复都稳定
- 用户能理解常见失败原因，不必靠读源码排障
- 仓库文档对当前能力和支持的工作流描述准确

## 架构演进方向

长期架构方向保持为：

- `src.engine.PersonalRecSys` 继续作为唯一公开主入口
- `src.pipeline.orchestrator.PipelineOrchestrator` 继续作为运行时编排中心
- `src.contracts` 逐步成长为真正的运行时契约层
- `src/evaluation` 保留为内部评测工具箱，不成为产品主链的运行时依赖

也就是说，已经验证过的能力应通过抽象或迁移回流到主链，而不是继续通过更多旁路脚本来堆功能。

## 近期执行优先级

当前推荐的阶段顺序是：

1. 补强推荐核心
2. 产品化 rerank 插槽
3. 标准化评测基础设施
4. 增加本地交互闭环
5. 做产品级工程打磨

如果必须做取舍：

- 推荐质量优先于界面打磨
- 主链产品化优先于继续扩张实验脚本数量
